# 理解验证与练习方法论

本文件定义阶段 3（提问验证）和阶段 4（练习）的执行逻辑。

---

## 阶段 3：提问验证

### 目标
确认用户真正理解了概念，而非只是"读过了"。核心方法：要求用户输出，而非选择。

### 根据概念类型选择验证策略

**概念类知识**（定义、原理、理论）
- 主问题："用你自己的话解释一下{概念}是什么？"
- 追问："能举一个具体的例子吗？"
- 验证点：能否脱离原文表述，用自己的语言重构

**机制类知识**（流程、算法、因果关系）
- 主问题："如果{条件变化}，结果会怎样？" / "{过程}的关键步骤是什么？"
- 追问："有没有什么情况下这个机制会失效？" / "边界情况是什么？"
- 验证点：能否推理而非背诵

**技能类知识**（操作、方法、工具使用）
- 主问题：给出一个具体问题，要求用户描述解决步骤或直接解决
- 追问："如果问题稍微变化（给出变体），你会怎么调整？"
- 验证点：能否迁移运用

### 追问深度策略

```
用户回答
  ├── 准确且有深度 → 赞同，补充一个延伸点，标记为 pass
  ├── 方向正确但不完整 → 用引导性问题补全，再确认一次
  │     ├── 补全后理解 → 标记为 pass
  │     └── 仍然困难 → 简要讲解缺失部分，标记为 partial
  └── 错误或无法回答 → 不直接纠正，而是：
        1. 指出矛盾点："你说的X，但根据刚才的学习，Y的情况下会怎样？"
        2. 给出提示引导重新思考
        3. 如果两次引导后仍困难 → 重新讲解关键点，标记为 fail
```

### 验证结果标记

| 结果 | confidence 增量 | 含义 |
|------|----------------|------|
| `pass` | +0.3 | 理解准确，能解释和举例 |
| `partial` | +0.1 | 方向正确但需要提示 |
| `fail` | +0.0 | 需要重新学习 |

### 开放讨论窗口（显式循环）

每个验证问题完成 AI 解析后，不要直接进入下一个问题。**必须进入以下显式循环**：

```
┌─────────────────────────────────────────────────┐
│  AI 完成解析 / 回答用户追问                       │
│         ↓                                        │
│  询问："还有想继续讨论的吗？还是进入下一题？"       │
│         ↓                                        │
│  用户回应                                        │
│    ├── 继续讨论（追问/质疑/延伸）→ AI 回应 → 回到顶部 │
│    └── 明确结束语 → 退出循环                      │
└─────────────────────────────────────────────────┘
```

**关键规则**：

1. **每次 AI 回复后都必须再次询问**是否继续讨论。绝不能在回答一轮追问后就自动跳到下一题
2. **退出条件**：只有用户说出明确的结束语时才退出循环，包括但不限于：
   - "没有了" / "可以了" / "下一题" / "继续" / "OK" / "好的下一个"
   - 用户主动表达的任何"结束讨论"意图
3. **不构成退出的回应**：用户的追问、反问、"那如果..."、"但是..."、"还有一个问题..."等都不构成退出，应继续讨论后再次询问
4. **自由对话模式**：讨论中不限制轮数，用户可以反复追问
5. **Confidence 加分**：讨论中产生的有价值洞察可酌情为 confidence 加分（最多额外 +0.1）
6. **记录讨论内容**：讨论中的所有对话轮次都需要在后续保存到验证对话记录中

## 阶段 4：练习

### 目标
通过实际应用巩固理解，将知识从"理解"转化为"能用"。

### 根据知识类型选择练习形式

**场景题（应用型知识）**
- 构造一个贴近实际的场景，要求用户应用所学概念解决
- 场景应比验证阶段更复杂，可能需要综合多个概念
- 示例："假设你正在{场景}，遇到了{问题}，你会如何运用{概念}来解决？"

**对比分析题（理论型知识）**
- 给出两个相关但不同的概念/方案/场景，要求分析异同
- 示例："{A}和{B}都可以用来解决{问题}，它们各自的优劣是什么？什么时候选哪个？"

**动手实践题（技能型知识）**
- 给出具体任务，要求用户写出解决方案/代码/步骤
- 示例："请写出一个{具体任务}的实现步骤" / "请完成以下{练习}"

### 出题规则

- 每个概念出 1-2 道练习题，不超过 2 道
- 如果阶段 3 验证为 `fail`，练习题降低难度，侧重基础巩固
- 如果阶段 3 验证为 `pass`，练习题可适当增加挑战性

### 练习后讨论窗口

每道练习题评估完成后，同样执行**与阶段 3 相同的显式讨论循环**（参见上方"开放讨论窗口（显式循环）"一节）。每次 AI 回复后都必须再次询问是否继续讨论，只有用户明确表示结束时才进入下一题或下一阶段。

### 练习评估

| 表现 | confidence 增量 | 处理 |
|------|----------------|------|
| 独立完成且正确 | +0.2 | 赞同，指出做得好的点 |
| 需要提示后完成 | +0.1 | 肯定思路，补充提示涉及的知识点 |
| 无法完成 | +0.0 | 讲解解题思路，标记概念需要加强 |

---

## 综合 Confidence 计算

阶段 3 和阶段 4 完成后，汇总 confidence：

```
最终 confidence = 基础值 + 阶段3增量 + 阶段4增量
```

- 新概念基础值 = 0.0
- 复习概念基础值 = 原 confidence × 0.8（衰减系数）

### 状态判定

| confidence 范围 | 状态 |
|----------------|------|
| ≥ 0.7 | `mastered` |
| 0.4 - 0.69 | `learning`（保持，下次复习） |
| < 0.4 | `weak`（标记需重点复习） |

---

## 交互原则

1. **鼓励为主**：即使回答错误，先肯定用户思考的努力，再引导纠正
2. **不直接给答案**：通过提问引导用户自己发现错误，而非直接纠正
3. **控制节奏**：每次只问一个问题，等用户回答后再继续
4. **适时总结**：阶段 4 结束时，简要总结本概念的核心要点和用户的掌握情况

## 验证对话记录

阶段 3 + 阶段 4 全部完成后，将验证和练习环节的完整对话整理保存。

### 保存路径

`domains/{domain}/discussions/{序号}-{概念名}.md`

序号与 `docs/` 中对应概念的文档序号一致。

### 文档格式

```markdown

domain: "{领域名}"
date: "{YYYY-MM-DD}"
result: "pass | partial | fail"
```
